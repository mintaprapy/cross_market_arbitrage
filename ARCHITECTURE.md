# Architecture Overview

## 总体思路

系统采用“模块化单体 + Ports and Adapters”架构：

- `domain` 只负责领域模型、换算公式、统计逻辑
- `application` 负责采集编排、状态机、告警评估
- `infrastructure` 负责外部市场数据和 SQLite
- `interfaces` 负责 FastAPI 与 Dashboard 展示
- `replay` 通过历史快照做事后回放分析，不参与实时主链路

这样可以保证：

- 核心业务逻辑不依赖具体交易所
- 数据源可替换
- Dashboard 不参与业务计算
- 原始数据和派生数据都可追溯

## 核心数据流

1. 获取 `USD/CNY`
2. 并发获取国内与海外报价
3. 将国内报价按公式换算为标准口径
4. 计算理论价差、可成交方向价差、滚动统计
5. 评估数据质量与阈值告警
6. 将告警分发给 notifier
7. 落库原始行情、快照、告警、通知投递记录
8. 通过 CLI / API / Web 对外展示

## 关键设计决策

- `监控层与执行层分离`：理论价差使用 `last`，可成交价差使用 `bid/ask`
- `原始数据与派生结果分离`：支持历史回放与重算
- `配置驱动`：监控组、税口径、阈值全部来自 YAML
- `滚动窗口按监控组独立`：避免跨品种污染统计结果
- `可降级 FX`：公开 FX 源失败时可退回配置的固定口径
- `通知与计算解耦`：notifier 作为独立适配器存在，失败只记录不阻塞主链路
- `回放只读历史快照`：分析模块不直接依赖实时内存状态

## 后续演进方向

- 将 FX 改成更实时的市场化来源
- 为每个数据源增加主备切换
- 扩展 Parquet / DuckDB 导出
- 若监控组显著增多，再拆出独立 worker / API 进程
