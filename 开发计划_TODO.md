# 跨市场商品价差监控开发计划 / TODO

## 1. 目标与范围

### 1.1 项目目标
- 实时采集国内期货、海外/链上映射标的、汇率数据。
- 将不同单位、币种、税口径统一成标准价格后进行比较。
- 输出价差、相对价差、滚动均值、标准差、z-score、价差变化等监控指标。
- 支持终端输出、历史落库、Web 看板、阈值告警。

### 1.2 首期覆盖范围
- 黄金：`AU` vs `XAUUSDT`
- 白银：`AG` vs `XAGUSDT`
- 铜：`CU / BC` vs `COPPERUSDT`
- 原油：`SC` vs `CL`（OKX / Hyperliquid / NYMEX 映射）

### 1.3 关键业务约束
- 监控层和执行层分离。
- 首期先做监控，不直接做下单执行。
- `AG`、`CU` 建议同时保留含税和去税口径。
- `SC` 与 `CL` 主要用于扩散/收敛监控，不假设强收敛到零。
- 执行价差后续必须基于 `bid/ask`，不能仅看 `last price`。

## 2. 建议技术路线

如无既有技术栈，建议优先用 Python 搭建首版：

- 运行方式：轮询版脚本起步，后续再演进到实时服务
- 数据处理：`pandas` / `numpy`
- 配置管理：`yaml` 或 `toml`
- 落库：`SQLite`
- Web 展示：`FastAPI + 前端页面` 或 `Streamlit`
- 告警：先控制台 + Web，后接飞书/Telegram/企业微信

这样可以先把“口径统一 + 价差计算 + 落库 + 可视化”快速跑通，再考虑低延迟优化。

## 4. 分阶段开发计划

### Phase 0：项目初始化
目标：搭出最小可运行骨架，明确配置与模块边界。

交付物：
- 项目目录结构
- 配置文件模板
- 统一数据模型
- 基础日志与错误处理

完成标准：
- 可以通过一个入口脚本启动程序
- 可以从配置文件读取监控组、轮询周期、阈值、数据源参数
- 日志中能区分采集、换算、监控、落库四类事件

### Phase 1：轮询版监控 MVP
目标：先跑通“采集 -> 标准化 -> 价差计算 -> 终端输出”。

交付物：
- 国内行情采集接口
- 海外/链上映射标的采集接口
- 汇率采集接口
- 标准化公式模块
- 实时指标计算模块
- 控制台表格输出

完成标准：
- 至少支持 1 组标的全链路跑通
- 可以计算 `normalized_px`、`spread`、`spread_pct`
- 支持设定轮询周期，例如 1s / 3s / 5s
- 异常数据不会导致主循环崩溃

### Phase 2：历史数据落库与回放
目标：支持保存原始行情与标准化结果，用于回测和统计分析。

交付物：
- SQLite 表结构
- 写库模块
- 数据导出能力（CSV）
- 滚动均值 / 标准差 / z-score 计算

完成标准：
- 程序运行时能持续写入 SQLite
- 可以按监控组查询历史价差
- 可以基于历史数据重算 z-score

### Phase 3：Web 看板与告警
目标：将实时监控从终端扩展到页面和消息通知。

交付物：
- 实时状态卡片
- 历史折线图
- 告警日志页
- 阈值告警模块

完成标准：
- 页面能看到各监控组的最新价差、z-score、更新时间
- 告警触发后可在页面中查看
- 支持基础消息推送预留接口

### Phase 4：可成交价差与盘口口径
目标：从“监控价差”升级到“可成交价差”。

交付物：
- `bid/ask` 数据采集
- 买入/卖出方向区分
- 可成交价差计算逻辑
- 数据质量校验规则

完成标准：
- 监控页面可切换 `last` 与 `bid/ask` 口径
- 可以区分“理论价差”和“可成交价差”
- 数据断流、过期、零值时能报警

### Phase 5：策略研究增强
目标：为后续量化研究和半自动交易提供分析基础。

交付物：
- 回归对冲比
- 波动率调整仓位
- 手续费 / 滑点 / 资金费率模型
- 历史回放分析报表

完成标准：
- 可以输出策略研究所需的基础特征
- 可以对不同标的组进行参数对比

## 6. 里程碑与优先级

### M1：首个可运行版本
- 范围：先支持 `AU_XAU`
- 能力：采集、换算、价差计算、终端输出
- 目标：验证全链路设计是否成立

### M2：多品种扩展
- 范围：增加 `AG_XAG`、`CU_COPPER`、`SC_CL`
- 能力：支持多监控组并行轮询
- 目标：验证配置驱动设计是否可复用

### M3：统计监控版本
- 范围：增加 SQLite 落库和 z-score
- 能力：历史回放、统计告警
- 目标：支持日内监控和策略研究

### M4：可视化版本
- 范围：增加 Web 看板和告警记录
- 能力：页面监控、筛选、追踪
- 目标：替代纯终端观察方式

### M5：研究增强版本
- 范围：加入盘口、成本、对冲比
- 能力：接近真实执行分析
- 目标：为后续策略模块做准备

## 9. 建议开发顺序

建议严格按下面顺序推进，避免一开始做得过重：

1. 先完成单组标的 `AU_XAU` 的轮询版 MVP。
2. 再把换算逻辑和监控逻辑抽象成可复用模块。
3. 接着补上 SQLite 落库和历史统计。
4. 然后再做 Web 看板和告警推送。
5. 最后加入盘口、成本模型和策略研究功能。

这样可以最快验证“数据是否可靠、价差是否有监控价值、架构是否可扩展”。
